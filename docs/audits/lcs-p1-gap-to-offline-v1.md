# LCS-P1 距离线下单机可用 V1 的缺口

审计日期：2026-07-04
审计基线：main@27c7b90
目标版本：第 1 版线下 Windows 单机可用

## 1. 总体判断

当前是否达到线下单机可用 V1：NO。

当前可以验收的部分：计算引擎规则、2026-04 验收计算、部分 DB workflow API、本地包生成脚本、备份恢复脚本文档化能力。

当前不能验收的部分：业务人员通过页面真实录入/导入、财务审核、资管申请、老板审批、HR 试算和导出完整闭环。

## 3. 核心阻塞项

### P1-1 真实数据入口闭环

必须补齐：

- `/commission/imports` 页面上传 Excel/CSV。
- 页面预览错误、行级校验、业务风险提示。
- 预览通过后提交到 DB，而不是内存 context。
- 员工、车辆、目标、车辆状态也要可写入对应表。
- 导入后对应台账页面必须能看到新增数据。

验收标准：业务人员不使用 Postman、不改代码、不调用命令行，也能完成一组账期数据导入。

### P1-2 台账 CRUD 和审核工作台

必须补齐：

- 订单新增/编辑/作废。
- 租金收入提交、财务审核/驳回。
- 外调利润提交、财务审核/驳回。
- 押金登记、退还状态更新、异常提示。
- 车辆状态流水新增。
- 指标调整申请、老板审批。

验收标准：销售、财务、资管、老板分别用页面完成各自动作，并形成审批日志。

### P1-3 HR 试算和老板审批页面动作

必须补齐：

- HR 从页面创建 Trial Run。
- HR 从页面发起试算/重算。
- 有 BLOCKER issue 时页面和服务层都阻止提交。
- 老板从页面审批/驳回 settlement run，驳回必须填原因。
- HR 能看到驳回原因并重算。
- approved run 才能从页面导出。

验收标准：完整跑通“试算 v1 -> 驳回 -> 修正 -> 试算 v2 -> 审批 -> 导出”。

### P1-4 本地数据库路径统一

必须补齐：

- `seed:acceptance`、Windows 脚本、health、web server 使用同一个默认 DB 路径。
- `setup-local.ps1` 或 reset 脚本能创建 health 所需的 local-data 子目录。
- `/api/health` 在标准初始化后返回 200。

验收标准：非开发人员按 Windows 使用说明操作后，健康检查通过。

### P1-5 权限从 header 模拟升级到本地可用登录

线下单机 V1 可以不做企业账号，但至少需要：

- 本地登录页。
- 用户选择/输入账号。
- 服务端 session/cookie。
- 页面动作按当前用户角色过滤。
- 销售只能看本人敏感台账，尤其押金。

验收标准：不依赖手工设置 `x-lcs-role` 请求头。

## 4. 最小补齐清单

建议不要直接进入演示数据包或部署增强，先做以下最小功能：

1. 统一 DB 路径和本地初始化脚本，保证 health 通过。
2. `/commission/imports` 真正支持文件上传、预览、提交 DB。
3. 台账页面改为 DB 列表，并提供订单/收入/外调/押金的新增入口。
4. 财务审核工作台：收入、外调利润审核/驳回。
5. 资管工作台：车辆状态登记、指标调整申请。
6. 老板工作台：指标调整审批、结算审批/驳回。
7. HR 工作台：创建 Trial Run、试算、提交审批、查看驳回、重算、导出。
8. Issue 和人工调整页面补齐创建/处理/审批动作。
9. 导出按钮绑定 approved run，并记录 export record。
10. 本地登录/session，替换页面操作中的 header 模拟。

## 5. 建议开发顺序

| 顺序 | 主题 | 为什么先做 | 验收方式 |
| --- | --- | --- | --- |
| 1 | DB 路径统一与本地初始化 | 这是单机可用基础 | `/api/health` 200，备份恢复可用 |
| 2 | 真实导入落库 | 当前最大阻塞是数据进不来 | Excel 导入 2026-05 后 DB 和页面可见 |
| 3 | 台账 DB 列表和手工录入 | 导入之外还要支持修正 | 页面新增/编辑后刷新可见 |
| 4 | 财务审核闭环 | 审核状态直接影响计算 | 审核前不计提，审核后计提 |
| 5 | 资管与指标调整闭环 | 目标金额影响达成率 | 未审批不影响，审批后影响 |
| 6 | HR 试算页面动作 | 让计算引擎进入业务流程 | 页面生成 run 和 snapshot |
| 7 | 老板审批/驳回页面动作 | 完成审批闭环 | 驳回后 run 保留，新 run 生成 |
| 8 | 导出页面动作 | 完成奖金发放输出 | xlsx 金额与页面/API 一致 |
| 9 | 本地登录和权限过滤 | 支撑真实多人角色试用 | 不同角色看到不同入口和数据 |

## 6. 线下单机 V1 验收标准

完成补齐后，V1 应能用一个脱敏真实账期验收：

1. 使用 Windows 脚本初始化环境和数据库。
2. 打开 `/api/health` 返回 200。
3. 用本地账号登录老板、销售、财务、资管、HR。
4. 通过页面导入或手工录入员工、车辆、周期、目标、订单、收入、外调利润、押金、车辆状态。
5. 财务审核收入和外调利润。
6. 资管提交指标调整，老板审批。
7. HR 创建 Trial Run 并生成试算 run。
8. 创建 BLOCKER issue 时阻止提交审批。
9. issue 关闭后 HR 提交审批。
10. 老板驳回，HR 修正/重算，新 runNo 不覆盖旧 run。
11. 老板审批新 run。
12. HR 导出奖金发放明细，导出记录绑定 approved run。
13. 备份 dev.db，再恢复备份，页面数据一致。
